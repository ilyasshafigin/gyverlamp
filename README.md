# GyverLamp

Здесь в проекте только прошивка и симулятор, схемы, инструкция по сборке в оригинальных репозиториях.

Цель проекта:

- разобраться в прошивках для микроконтроллеров и C++
- структурировать файлы проекта, отвязаться от скетчей Arduino
- сделать так, чтобы можно было легко добавлять новый функционал

Ссылки на аналогичный проекты, идеями которых вдохновлен, также по этим же ссылкам схемы сборки лампы:

- [GyverLamp2](https://github.com/AlexGyver/GyverLamp2) от AlexGyver
- [GyverLamp](https://github.com/Whilser/GyverLamp) от Whilser
- [GyverLamp](https://github.com/gunner47/GyverLamp) от gunner47
- [FieryLedLamp](https://github.com/MishanyaTS/FieryLedLamp) от MishanyaTS

## Особенности

### Железо

- Контроллер ESP8266 в лице платы NodeMCU или Wemos D1 mini
- Гибкая адресная матрица 16×16 WS2812B RGB
- Сенсорная кнопка на TTP223
- Микрофон MAX9814, без конденсатора между OUT и A0

### Прошивка

- PlatformIO + Arduino
- Больше 20 отобранных эффектов, настройка яркости и параметров каждого эффекта, плавное переключение
- Палитра, большенство эффектов работает с ней
- Режим ротации эффектов: последовательно или случайно
- Поддержка микрофона для модуляции яркости и других параметров эффектов, специальный эффект Equalizer
- Упор на MQTT + Home Assistant
- Web UI
- Поддержка некоторых мобильных приложений, управление по UDP протоколу
- Таймер автоотключения
- Уведомления: системные (WiFi, MQTT, OTA) и пользовательские, возможность заглушить их в ночное время
- Все работает на последних версиях библиотек, но на всякий случай версии зафиксированы
- Основные зависимости:

  - `fastled/FastLED` - работа со светодиодной матрицей
  - `GyverLibs/Settings` - web ui
  - `paulino/HaMqttEntities` - MQTT + Home Assisntant
  - `gyverlibs/EncButton` - работа с сенсорной кнопкой

### Симулятор

- Браузерный WASM-симулятор эффектов: смотреть и настраивать в браузере без лампы, прошивки и ESP8266.
- Выбор эффекта и палитры, яркость/скорость/масштаб/FPS, пауза и покадровый шаг, diffuser-превью под белый плафон.
- Онлайн: https://ilyasshafigin.github.io/gyverlamp/
- Локальный запуск описан в разделе «Разработка».

### Корпус

- [3д печатный корпус](https://3dpm.ru/open_projects/GL1.1/) от Fattum

### WiFi

- Если сохранённых настроек WiFi нет, лампа запускает точку доступа для настройки: `GyverLamp` / `12345678`, адрес `192.168.4.1`.
- Если настройки есть, лампа пытается подключиться к домашнему WiFi около 10 секунд.
- При неудачном подключении лампа открывает точку доступа для настройки и продолжает пытаться переподключиться каждые 5 секунд.
- После успешного подключения точка доступа выключается, IP-адрес лампы выводится в Serial и может быть показан на матрице 5 нажатиями кнопки.
- Если включить лампу 5 раз подряд с быстрыми перезагрузками, сохранённые WiFi-настройки сбрасываются, чтобы можно было заново открыть режим настройки.
- Точка доступа автоматически выключается через 5 минут, если к ней никто не подключился.

### Кнопка

- 1 нажатие — включить/выключить лампу; если активно пользовательское уведомление — выключить его.
- 2 нажатия — следующий эффект.
- 3 нажатия — предыдущий эффект.
- 5 нажатий — показать IP-адрес лампы на матрице.
- Удержание — плавно менять яркость текущего эффекта; каждое новое удержание меняет направление регулировки.

### Уведомления

- Системные уведомления показываются поверх эффекта и слегка затемняют фон.
- WiFi: подключение — мягкий синий пульс, успешное подключение — короткий зелёный пульс, ошибка — оранжевый пульс около 10 секунд.
- MQTT: подключение — фиолетовый пульс, успешное подключение — короткий бирюзовый пульс, ошибка — янтарный пульс около 10 секунд.
- OTA: во время прошивки — бирюзовый прогресс/бегущий огонёк сверху, успех — зелёный пульс, ошибка — красный двойной пульс.
- Пользовательские уведомления включаются через Home Assistant/MQTT: выбрать `Уведомление` (`Warning` или `Alarm`) и при необходимости задать `Длительность уведомления` в секундах.
- `Warning` — оранжевый пульс всей лампы, `Alarm` — быстрый красный пульс всей лампы. `Off` выключает пользовательское уведомление.
- Можно заглушить уведомления в заданный интервал, когда питание выключено. Тревога, предупреждения и OTA-статусы продолжают показываться.

## Разработка

### Как запустить проект локально

1. Склонировать репозиторий:

    ```bash
    git clone <url-репозитория>
    cd gyverlamp
    ```

2. Установить PlatformIO, если он ещё не установлен. Например, через VS Code расширение PlatformIO IDE или через CLI:

    ```bash
    pip install platformio
    ```

    Для коротких команд можно также установить [`just`](https://github.com/casey/just).

3. Создать локальный конфиг из примера:

    ```bash
    cp platformio.local.example.ini platformio.local.ini
    ```

4. Отредактировать `platformio.local.ini` под свою лампу:

   - `AP_SSID` — имя WiFi-точки лампы в режиме настройки;
   - `CURRENT_LIMIT` — лимит тока блока питания в миллиамперах;
   - `upload_port` — IP-адрес лампы для OTA-прошивки, только после первой настройки WiFi, если IP ещё неизвестен;
   - при необходимости добавить свои env'ы для нескольких ламп.

    `platformio.local.ini` не попадает в git, поэтому в нём можно держать локальные IP-адреса и настройки конкретного железа.

    Публичный `platformio.ini` содержит только общие настройки проекта. Все настройки конкретной лампы находятся в `platformio.local.ini`.

5. Собрать прошивку:

    ```bash
    pio run -e lamp1_usb
    # или
    just build lamp1_usb
    ```

6. Первая прошивка обычно делается по USB:

    ```bash
    pio run -e lamp1_usb -t upload
    # или
    just upload lamp1_usb
    ```

    После этого лампа поднимет точку доступа, где можно настроить WiFi и MQTT.

7. Настроить лампу

    При первом запуске лампа создает точку доступа, по умолчанию это `GyverLamp`, пароль `12345678`.
    Нужно подключиться к ней, открыть адрес "192.168.4.1" и ввести название и пароль локального WiFi, сохранить и перезагрузить.

    После перезагрузки лампа подключится к WiFi, роутер выдаст ей IP, по этому IP можно будет продолжить настройку. Рекомендуется зафиксировать этот IP за ней, чтобы можно было обновляться по OTA.
    Дальше можно настроить MQTT для её управления из Home Assistant. Открыть новый IP-адрес лампы в браузере, и ввести данные в разделе MQTT.

8. Последующие прошивки можно делать по OTA, если в `platformio.local.ini` указан правильный `upload_port`:

    ```bash
    pio run -e lamp1_ota -t upload
    # или
    just upload lamp1_ota
    ```

    Если в локальном конфиге задан `default_envs`, то env можно не указывать:

    ```bash
    pio run
    # или
    just build
    ```

    Аналогично, `just upload` прошивает default env'ы, а `just build lamp1_ota lamp2_ota` / `just upload lamp1_ota lamp2_ota` выполняет команду для нескольких env'ов по очереди.

### Локальный симулятор эффектов

Симулятор живёт в `sim/` (подробнее: [`sim/README.md`](sim/README.md)). Запуск локально:

```bash
cd sim/web
npm install
npm run sim
# или из корня проекта
just run sim
```

`npm run sim` сначала собирает WASM-артефакты, потом поднимает статический сервер на http://localhost:8080.

### LSP / `compile_commands.json` (для агентов)

LSP (clangd) использует `compile_commands.json` в корне проекта.  
Он не лежит в репозитории и пересоздаётся вручную.

Сгенерировать для default env (`lamp1_ota`):

```bash
pio run -e lamp1_ota -t compiledb
```

Если LSP не видит Arduino/ESP8266 или библиотечные хидеры — настройки `extra_script.py` уже включают toolchain-инклуды (`COMPILATIONDB_INCLUDE_TOOLCHAIN=True`).

## Проблемы

### Подтормаживание анимации эффектов

Иногда эффекты могут идти рывками: анимация замирает на доли секунды, появляются микролаги, мерцание или сбои при обновлении матрицы. Для связки ESP8266 + WS2812B это типичная проблема на границе таймингов, WiFi и фоновых сетевых задач.

Возможные причины:

- WS2812B требует очень точный тайминг сигнала.
- На ESP8266 WiFi-прерывания могут ломать bit-bang-передачу данных на ленту.
- У FastLED есть известная проблема для ESP8266 + WiFi: иногда нужен повторный `show()`.
- MQTT или Home Assistant могут забирать слишком много времени в основном цикле и мешать равномерной отрисовке эффектов.

Что помогло в моем случае:

1. Удалить устройство лампы из MQTT-интеграции Home Assistant.
2. Перезагрузить MQTT-интеграцию.
3. Дождаться, пока устройство появится снова.

После этого лаги ушли: видимо, MQTT/HA перестали перегружать обмен с лампой.

Другие варианты, которые можно попробовать:

- Проверить версию платформы ESP8266. Arduino core 3.x часто ведет себя хуже с FastLED на ESP8266. В некоторых проектах помогает откат на core 2.7.4 / `platform = espressif8266@2.6.3`.
- Попробовать запретить прерывания FastLED во время вывода на матрицу:

    ```cpp
    #define FASTLED_ALLOW_INTERRUPTS 0
    ```

- Или подобрать количество повторов после прерывания:

    ```cpp
    #define FASTLED_INTERRUPT_RETRY_COUNT 1
    ```

Эти флаги нужно задавать до подключения FastLED. После изменения платформы или флагов нужно пересобрать прошивку и проверить поведение на реальной лампе.
